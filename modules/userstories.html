Module - User Stories

Wat zijn User Stories?

Bij de module user stories behandel je verhalen van potenciÃ«le gebruikers van jouw app of website.

Het is belangrijk dat je je kunt inleven in de mogelijke bezoekers en je website of app, hiermee kun je op basis daarvan kunt aanpassen of prioriteiten kunt stellen voor nieuwe functies.

Een gemiddelde user story gaat als volgt: Je beschrijft wie je user is, wat hij/zij wil kunnen doen op je app/site en waarom deze persoon dit wil.

Vervolgens heb je acceptatie criteria, dit zijn de dingen die gedaan kunnen worden voor die user om tevreden te zijn met je implementatie.

Hoe dit helpt

Je kunt user stories zelf bedenken, ontvangen van een opdracht gever of van echte users zelf. Met deze stories te weten komen welke functies veel gewilt zijn.

Om te bepalen hoe je deze user stories kunt gebruiken werken we met de MoSCoW analyse. Bij deze methode deel je de user stories in op hoe essencieel ze zijn om te gaan maken. Je deelt de stories op in Must-Haves, Should-Haves, Could-Haves en Won't-Haves.

Door de stories op te delen in stukjes op basis van hoe belangrijk ze zijn, kun je inschatten waar je eerst aan moet werken en heeft de klant er toch iets aan als er geen tijd of geld meer over is om verder te werken.

Voorbeeld user stories

Dit is een voorbeeld van een gemiddelde user story voor een supermarkt app:

Als: klant,

Wil ik: aanbiedingen kunnen bekijken,

Omdat: ik dan weet welke producten mij geld besparen

Acceptatie criteria:

  • Er is een scherm waar ik de aanbiedingen van vandaag kan zien.
  • Kunnen zien welke producten er afgeprijst zijn, de nieuwe prijs en hoe veel procent goedkoper dit is.

De MoSCoW analyse

Bij de MoSCoW analyse deel je geven user stories op in 4 delen:

  • Must-haves: Funcies die absoluut nodig zijn om de site/app te laten werken.
  • Should-Haves: Optionele functies die toch belangrijk zijn, deze kun je gaan implementeren als de Must-Haves klaar zijn.
  • Could-Haves: Features die niet zo belangrijk zijn, maar wel toegevoed kunnen worden als er genoeg tijd/geld over is.
  • Won't-Haves: Dingen die niet op de website komen, dit kan zijn door gebrek aan tijd of geld, maar ook omdat de feature helemaal niet nodig is of onhandig zou zijn.
